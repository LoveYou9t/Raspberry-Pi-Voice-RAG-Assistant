# 服务启动文档（简版）

本文档用于快速启动当前语音 RAG MVP 服务。

## 1. 启动前准备

1. 已安装 Docker Desktop，并确保 Docker 正常运行。
2. 当前目录为项目根目录。
3. 如需调整模型或参数，可先修改 .env 文件。

## 2. 首次启动（推荐）

1. 在项目根目录执行：
   docker compose up -d --build
2. 查看服务状态：
   docker compose ps
3. 首次启动会自动预热默认模型（`llama3.2:3b` 与 `whisper.cpp q5_0`，失败时回退 `faster-whisper`），耗时更长。
4. 正常状态为：
   edge_ollama、edge_fastapi、edge_frontend 为 Up；
   edge_ollama_init、edge_stt_init 为 Exited (0)。

## 3. 访问地址

1. 前端页面：<http://localhost:8080>
2. 后端健康检查：<http://localhost:8000/healthz>

## 3.1 WebSocket 音频协议（当前默认）

1. 传输格式：WebSocket 二进制音频帧 + 文本控制帧（JSON）。
2. 上行音频：默认 Opus（浏览器不支持时自动回退 PCM16），采样率默认 16000Hz。
3. 下行音频：默认 Opus，后端会按客户端声明的目标采样率自动重采样后下发。
4. 客户端连接后会自动发送 `client_audio_config`，服务端返回 `audio_config_applied` 确认实际生效配置。
5. 若运行环境缺少 Opus 能力（浏览器或后端依赖不可用），系统会自动回退到 PCM16 流程，业务可继续运行。

## 4. 常用运维命令

1. 查看实时日志：
   docker compose logs -f
2. 重启全部服务：
   docker compose restart
3. 停止并删除容器（保留数据卷目录）：
   docker compose down

## 5. 模型检查（自动预热后）

1. 查看 Ollama 模型：
   docker exec edge_ollama ollama list
2. 查看后端健康信息：
   curl <http://localhost:8000/healthz>
   重点关注 llm 字段：
   - llm.warmup.ok / llm.warmup.message
   - llm.keepalive.running / llm.keepalive.last_keepalive_at
3. 若 STT 预期使用 5bit，确认以下路径存在：
   主机：whisper_cache/models/ggml-small-q5_0.bin
   容器：/app/model_cache/models/ggml-small-q5_0.bin
4. 若 llama3.2:3b 未就绪，优先检查：
   先确认你使用的是哪种导入方式：
   - GGUF 本地创建：`ollama_models/` 下是否存在 `.env` 的 `OLLAMA_LOCAL_MODEL_PATH` 对应文件（文件名需完全一致）。
   - Ollama 标准结构（blobs/manifests）：应放在 `ollama_data/` 下（运行目录）；新版也会尝试从 `ollama_models/.ollama` 自动同步。
   若都不满足，则需要目标机器可联网执行 `ollama pull`。

## 6. 简单排查

1. 前端打不开：确认 edge_frontend 容器为 Up，并检查 8080 端口未被占用。
2. 后端无响应：确认 edge_fastapi 为 Up，执行 docker compose logs -f fastapi_backend 查看错误。
3. LLM 无返回：确认 edge_ollama 为 Up，并执行 docker exec edge_ollama ollama list 检查模型是否存在。
   若模型存在但首次请求仍慢，可执行：
   docker exec edge_ollama ollama ps
   并结合 healthz 的 llm.keepalive 字段判断是否在保活。
4. 模型已复制但仍不可见：
   - 复制的是 GGUF 文件时，检查 `.env` 的 `OLLAMA_LOCAL_MODEL_PATH` 是否与实际文件名完全一致。
   - 复制的是 Ollama 标准目录时，优先放到 `ollama_data/`；若放在 `ollama_models/.ollama`，查看 `docker compose logs -f ollama_init` 是否有 sync 成功日志。
5. 无语音播放：先看浏览器是否允许麦克风权限，再检查后端日志中的 WebSocket 连接信息。
6. 首次启动很慢：可查看初始化容器日志判断是否仍在拉模型：
   docker compose logs -f ollama_init
   docker compose logs -f stt_init
7. 若提示 `ollama_init ... exit 1`：
   拉取最新代码后重建（新版已将预热失败改为重试并不阻断启动）：
   git pull --rebase
   docker compose down --remove-orphans
   docker compose up -d --build
8. 若提示 `stt_init ... exit 1`：
   拉取最新代码后重建（新版已将 stt_init 改为非阻断兜底执行）：
   git pull --rebase
   docker compose down --remove-orphans
   docker compose up -d --build
9. 若 `ollama_init` 日志出现 `Error: unknown command "/bin/sh" for "ollama"`：
   这通常是旧版编排未覆盖 `ollama/ollama` 默认 entrypoint 导致。请更新到最新版 `docker-compose.yml` 后重建：
   git pull --rebase
   docker compose up -d --build
   然后再次检查：
   docker compose logs -f ollama_init

## 8. LLM 保活与预读取配置

在 .env 中可设置：
1. LLM_KEEP_ALIVE=24h
2. LLM_WARMUP_ON_STARTUP=1
3. LLM_WARMUP_PROMPT=你好
4. LLM_WARMUP_TIMEOUT_SECONDS=30
5. LLM_WARMUP_RETRIES=2
6. LLM_WARMUP_RETRY_DELAY_SECONDS=2
7. LLM_KEEPALIVE_ENABLED=1
8. LLM_KEEPALIVE_INTERVAL_SECONDS=300
9. LLM_KEEPALIVE_PROMPT=嗯

## 8.1 WebSocket 音频协商配置（可选）

在 .env 中可设置：
1. WS_DEFAULT_AUDIO_CODEC=opus
2. WS_DEFAULT_UPLINK_SAMPLE_RATE=16000
3. WS_DEFAULT_DOWNLINK_SAMPLE_RATE=16000
4. WS_SUPPORTED_AUDIO_CODECS=opus,pcm16
5. WS_SUPPORTED_SAMPLE_RATES=16000,24000,48000
6. WS_OPUS_FRAME_MS=20
7. WS_OPUS_BITRATE=24000

## 7. 目录提示

1. 知识库目录：knowledge_base
2. 向量数据目录：lancedb_data
3. 模型数据目录：ollama_data（Ollama 运行目录，适合放标准 blobs/manifests 结构）
4. 离线 GGUF 目录：ollama_models（仅用于 `OLLAMA_LOCAL_MODEL_PATH` 的 GGUF 文件输入）

将新的 .md 或 .txt 放入 knowledge_base 后，可在后端容器内运行知识同步脚本更新索引。
